# Santiago's Second Brain — Project Context

## What This Is

A complete context file for building Santiago's personal AI second brain system. This captures everything discussed across the design session so an OpenClaw harness can be set up with full context.

## 1. The Vision

Build a personal second brain that compounds knowledge over time. Inspired by Garry Tan's GBrain system (17,888 pages, 34 skills, 21 crons) but adapted to Santiago's scale and constraints. The goal is not to replicate GBrain — it's to steal the 5 patterns that produce 80% of the value and skip the infrastructure that serves Garry's scale (YC CEO, hundreds of founder meetings/week).

### Core Patterns Borrowed from GBrain

1. **Compiled truth + timeline schema** — Every page has current understanding on top (rewritten when evidence changes), append-only timeline below (never edited, only added to).
2. **Signal detection** — Capture entities + insights from every interaction automatically.
3. **Entity propagation** — After any interaction, update ALL related pages (people, businesses, meetings mentioned).
4. **Tiered enrichment** — Stub on first mention (Tier 3) → richer after 3+ mentions (Tier 2) → full page after meeting/deep interaction (Tier 1).
5. **Daily compilation** — Nightly cycle: synthesize the day's inputs, fix stale pages, surface connections, consolidate.

### What We Explicitly Skip

- Postgres/PGLite/Supabase (Obsidian is the database)
- Vector embeddings and hybrid search (not needed at this scale — yet)
- MCP server, OAuth, HTTP transport
- Minions job queue
- BrainBench eval harness
- Voice-to-brain pipeline
- Multi-machine federation
- Knowledge graph with typed SQL links
- Storage tiering
- 34 skills (we need ~8-10)

## 2. Architecture

```

  ↕ iCloud sync
```

### Role of Each Component

| Component | Role | Access |
|-----------|------|--------|
| **OpenClaw** | The harness. Runs crons, processes inputs, writes pages, propagates entities, compiles daily digests. Always-on. | Read + Write to vault |
| **Obsidian Vault** | The brain. Single source of truth. All knowledge lives here as markdown. Syncs via iCloud to phone. | Readable by all, writable by OpenClaw + Santiago |
| **Claude Code** | Coding tool + manual work. Reads vault and context files for session awareness. Does NOT run autonomously. | Reads vault. Writes code, not brain pages. |
| **Judge** | Existing harness with deep context about Santiago. Consulted for specific tasks. | TBD — needs clarification on what Judge is and what context it holds |

### Key Constraint

Claude Code cannot be the harness — it's session-based, can't run crons or loops. OpenClaw handles all autonomous/scheduled work. Claude Code reads a context file that OpenClaw generates so it has session awareness.

## 6. Skills to Build

### Skill Architecture Overview

Skills follow the GBrain SKILL.md format:
```yaml
---
name: skill-name
version: 1.0.0
description: |
  What it does, when to use it.
triggers:
  - "trigger phrase"
tools:
  - exec
  - read
  - write
mutating: true|false
---
```

Body includes: **Contract** (what it guarantees), **Phases** (step-by-step), **Output Format** (what it produces).

### The Skills

#### Capture Skills (get data into the vault)

**1. `claude-extract`**
- **Purpose:** Extract decisions, entities, insights from Claude Code sessions and Claude.ai memory files
- **Input:** Claude Code session transcripts, Claude.ai local memory files
- **Output:** Updates to people pages, business pages, decision logs in vault
- **Trigger:** Daily cron + manual "extract from claude"
- **Reads:** Claude.ai memory files on disk, Claude Code session data
- **Writes:** People/, Businesses/, Daily/

**2. `meeting-ingest`**
- **Purpose:** Process transcribed meeting recordings into structured vault pages with entity propagation
- **Input:** Transcribed meeting text (from iCloud recordings, transcribed via existing skill)
- **Output:** Meeting page in Meetings/, updated People/ pages for all attendees
- **Trigger:** Manual "process this meeting" or cron watching for new transcriptions
- **Reads:** Transcription output, existing People/ pages
- **Writes:** Meetings/, People/

**3. `media-ingest`**
- **Purpose:** Process YouTube videos, X threads, articles into vault pages
- **Input:** URL or content
- **Output:** Source material page with personal elaboration, entity extraction
- **Trigger:** Manual "ingest this" or cron watching a queue
- **Reads:** URL content (transcribe video, fetch article)
- **Writes:** 1 - Source Material/, People/ (if people mentioned)

**4. `book-mirror`**
- **Purpose:** Create chapter-by-chapter personal mirror of a book. Left column: what the author says. Right column: how it maps to Santiago's specific life using soul.md, personality.md, business context, blind spots.
- **Input:** Book content (PDF, chapter notes, or manual chapter-by-chapter feed)
- **Output:** Mirror page in Mirrors/ with two-column analysis per chapter
- **Trigger:** Manual "mirror this book"
- **Reads:** soul.md, personality.md, Businesses/, existing Mirrors/ for cross-book connections
- **Writes:** Mirrors/

#### Processing Skills (connect and enrich)

**5. `entity-propagate`**
- **Purpose:** After any new page is created or updated, walk through all mentioned people/companies and update their pages with the new reference. Create stub pages (Tier 3) for first-time mentions.
- **Input:** Any newly written/updated vault page
- **Output:** Updated People/ and Businesses/ pages with new timeline entries
- **Trigger:** Post-write hook — runs automatically after any skill writes a page
- **Reads:** The new/updated page, existing People/ and Businesses/ pages
- **Writes:** People/, Businesses/

**6. `enrich`**
- **Purpose:** Upgrade people/company pages based on mention frequency. Tier 3 (stub) → Tier 2 (basic context from available sources) → Tier 1 (full page with web research, relationship context, meeting history).
- **Input:** Person or company slug
- **Output:** Enriched page with compiled truth
- **Trigger:** Automatic when mention count crosses threshold (3 → Tier 2, meeting/8+ mentions → Tier 1), or manual "enrich [person]"
- **Reads:** All vault pages mentioning the entity, web sources
- **Writes:** People/, Businesses/

#### Output Skills (get data out / compile)

**7. `daily-compile`**
- **Purpose:** Nightly dream cycle. Review all changes to vault from the day, synthesize connections, update stale compiled truths, flag orphan pages, write daily digest.
- **Input:** All vault changes from the past 24 hours
- **Output:** Daily/ page, updated compiled truths across affected pages
- **Trigger:** Nightly cron (11 PM)
- **Reads:** All recently modified vault pages, soul.md for context
- **Writes:** Daily/, updates to any stale pages

**8. `context-for-claude-code`**
- **Purpose:** Generate a context file that Claude Code reads at session start. Contains: active projects, recent decisions, open threads, relevant people context, what changed in the vault recently.
- **Input:** Current vault state
- **Output:** A context markdown file at a known location Claude Code can read
- **Trigger:** Morning cron (6 AM) + on-demand
- **Reads:** Daily/, People/, Businesses/, open threads across vault
- **Writes:** Context file (location TBD — could be vault root or a separate path)

### Skill Dependency Graph

```
claude-extract ──→ entity-propagate ──→ enrich (if threshold crossed)
meeting-ingest ──→ entity-propagate ──→ enrich
media-ingest ───→ entity-propagate ──→ enrich
book-mirror ────→ entity-propagate

daily-compile (reads all changes, writes Daily/ + updates stale pages)
context-for-claude-code (reads vault, writes context file)
```

### Proposed Cron Schedule

| Time | Skill | What |
|------|-------|------|

| Manual/queued | `media-ingest` | Process YouTube/X/articles from a to-process list |
| Manual | `book-mirror` | Create book mirror on demand |
| Post-write | `entity-propagate` | Auto-fires after any skill writes a page |
| Threshold | `enrich` | Auto-fires when mention count crosses tier threshold |
| 6:00 AM | `context-for-claude-code` | Generate morning context file |

## 9. Reference: GBrain Skill Format (SKILL.md)

```yaml
---
name: skill-name
version: 1.0.0
description: |
  One paragraph. What it does, when to use it.
triggers:
  - "trigger phrase users actually say"
tools:
  - exec
  - read
  - write
mutating: true|false
---
```

### Required Sections
- **Contract** — what the skill guarantees
- **Phases** — step-by-step execution
- **Output Format** — what it produces

### Quality Checklist (11 items)
1. SKILL.md with frontmatter + contract + phases
2. Deterministic code (scripts/*.ts) if applicable
3. Cross-modal eval (3 frontier models, 3 providers) — informational
4. Unit tests
5. Integration tests
6. LLM evals
7. Resolver trigger entry
8. Resolver eval (triggers route correctly)
9. Check-resolvable (DRY + MECE audit)
10. E2E smoke test
11. Brain filing (if it writes pages)

### Anti-Patterns
- Writing tests before cross-modal eval (locks in mediocrity)
- Budget models for eval (C student grading A student)
- Single provider for all eval slots (correlated blind spots)
- Code with no SKILL.md (invisible to resolver)
- Two skills doing the same thing (merge or kill one)

## 10. Garry Tan's Article — Key Takeaways Applied to Santiago

### What Garry Built
- 100,000-page structured knowledge base
- 100+ skills, 100 daily crons
- Every person, company, meeting, book gets a page
- Entity propagation: after every meeting, update ALL related pages
- Book mirrors: chapter-by-chapter with personal mapping
- Meeting prep: pulls full context on every attendee before meetings
- Skillify loop: manual workflow → extract pattern → reusable skill → compound

### Garry's Core Insight
"The difference between a filing cabinet and a nervous system. The filing cabinet stores things. The nervous system connects them, flags what's changed, and surfaces what's relevant to right now."

### Santiago's Adaptation
Same principles, 1/10th the infrastructure. Santiago doesn't meet hundreds of founders/week. He runs 4-6 businesses with ~50 key relationships. The compounding works at his scale too — every meeting enriches the next prep, every book mirror is richer because the brain knows more, every decision is better because the context is fuller.

The key difference: Garry has a dedicated engineering team and runs OpenClaw + Hermes as production deployments. Santiago has one Mac mini, one OpenClaw instance, Claude Code for manual work, and an Obsidian vault. The system must be simple enough that one person maintains it.
